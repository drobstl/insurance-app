# AFL Upload + Extraction Failure Taxonomy

| Failure Type | Detection Signal (what in the code tells you this happened) | Automatic Fallback Behavior | User-Facing Message | Owner (file path) |
|---|---|---|---|---|
| Signed URL generation fails | `POST /api/ingestion/v3/upload-url` returns non-OK or throws timeout (`Upload URL generation timed out.` in Add Client flow). | Add Client: fallback to direct parse only on timeout path; ApplicationUpload: no direct fallback from signed URL failure. **NOT IMPLEMENTED — needs Phase 2** (consistent fallback in both entry points). | "Upload couldn't start. Please try again." | `web/app/api/ingestion/v3/upload-url/route.ts`, `web/app/dashboard/clients/page.tsx`, `web/components/ApplicationUpload.tsx` |
| Storage PUT fails / times out | Signed URL `PUT` returns non-2xx (`Upload failed (...)`) or timeout (`Upload timed out while sending file.`). | Add Client: fallback only for specific signing strings or timeout; no retry on general network failure. ApplicationUpload: retries up to 3 times on transient `TypeError`, then fails. **NOT IMPLEMENTED — needs Phase 2** (unified retry + fallback). | "File didn't upload. Please try again." | `web/app/dashboard/clients/page.tsx`, `web/components/ApplicationUpload.tsx` |
| Cloud Tasks queue enqueue fails | `enqueueIngestionV3ProcessJob(...)` throws in jobs API; route returns `TASK_ENQUEUE_FAILED` and writes failed status. | Current behavior returns API error; no transparent direct-parse fallback in this route. **NOT IMPLEMENTED — needs Phase 2** (auto-direct-parse or forced immediate processor path). | "We couldn't queue processing. We're retrying now." | `web/app/api/ingestion/v3/jobs/route.ts`, `web/lib/cloud-tasks.ts` |
| Queue job stalls (stuck in queued/processing too long) | Status polling sees `queued/uploading/processing` beyond stall cutoff (`V3_QUEUE_STALL_FALLBACK_MS` in Add Client; no dedicated queue-stall fallback in `ApplicationUpload`). | Add Client triggers direct parse after stall threshold; ApplicationUpload waits to overall timeout and then errors. **NOT IMPLEMENTED — needs Phase 2** (identical stall fallback in both paths). | "Processing your file now..." | `web/app/dashboard/clients/page.tsx`, `web/components/ApplicationUpload.tsx` |
| Processor endpoint fails / times out | `/api/ingestion/v3/jobs/[jobId]/process` returns `INTERNAL_ERROR` or worker classifies terminal error (processor catches and sets failed status). | Current behavior depends on polling client: Add Client falls back to direct parse for `INTERNAL_ERROR`/`CLAUDE_SCHEMA_INVALID`; ApplicationUpload does the same for those codes only. Timeout at processor layer may still bubble as generic failure. **NOT IMPLEMENTED — needs Phase 2** (uniform fallback map). | "We couldn't fully read this file. Please review missing fields." | `web/app/api/ingestion/v3/jobs/[jobId]/process/route.ts`, `web/lib/ingestion-v3-processor.ts`, `web/app/dashboard/clients/page.tsx`, `web/components/ApplicationUpload.tsx` |
| Claude extraction returns incomplete data (core fields missing) | Extraction succeeds (`review_ready/saved`) but returned `application.data` has missing/null core fields; there is no hard gate enforcing core completeness in API contract. | Current behavior accepts payload as-is. No automatic second pass by missing-core-fields policy. **NOT IMPLEMENTED — needs Phase 2** (partial-success contract + confidence + optional enrichment pass). | "Some fields need your review." | `web/lib/application-extractor.ts`, `web/lib/ingestion-v3-processor.ts`, `web/app/dashboard/clients/page.tsx`, `web/components/ApplicationUpload.tsx` |
| Claude extraction returns malformed/unparseable response | `application-extractor` throws `Failed to parse AI response.` / `No response received from AI.`; processor classifies `CLAUDE_SCHEMA_INVALID` or retryable request failures. | Add Client and ApplicationUpload fallback to direct parse only for certain codes. For direct parser route itself, malformed output can still return 500 with raw error text. **NOT IMPLEMENTED — needs Phase 2** (consistent one-retry + human-safe message map). | "We had trouble reading this file. Please review." | `web/lib/application-extractor.ts`, `web/lib/ingestion-v3-processor.ts`, `web/app/api/parse-application/route.ts` |
| Stale idempotency match (job exists but stuck/failed/old) | Jobs API always reuses idempotency match when found (`findExistingIngestionV3JobByIdempotency` then immediate return), without stale/failed age checks. | No stale guard yet. **NOT IMPLEMENTED — needs Phase 2** (always fresh parse per locked decision, or explicit stale policy). | No user message needed (transparent). | `web/app/api/ingestion/v3/jobs/route.ts`, `web/lib/ingestion-v3-store.ts` |
| Mixed status-poll timeout (ambiguous state during polling) | Poll request timeout or non-JSON status response (`Unexpected response while checking parse status...`). | Add Client now falls back on timed-out status calls; ApplicationUpload surfaces timeout error and stops. **NOT IMPLEMENTED — needs Phase 2** (uniform fallback for ambiguous poll state). | "We lost track of processing status. Please try again." | `web/app/dashboard/clients/page.tsx`, `web/components/ApplicationUpload.tsx`, `web/app/api/ingestion/v3/jobs/[jobId]/route.ts` |
| Fallback parser itself times out or errors | `/api/parse-application` request times out (`Direct parser request timed out.`) or returns non-success (`Direct parser failed (...)`). | No secondary fallback after direct parser failure. **NOT IMPLEMENTED — needs Phase 2** (clear terminal partial/error policy and optional retry). | "We couldn't read this file right now. Please try again or enter fields manually." | `web/app/dashboard/clients/page.tsx`, `web/components/ApplicationUpload.tsx`, `web/app/api/parse-application/route.ts` |
| Spinner state exceeds timing SLA (loading beyond 90s) | Add Client loop budget is `IMPORT_PARSE_TIMEOUT_MS = 120_000` then fallback parse (can exceed 90s end-to-end); ApplicationUpload uses separate timeout budget and can end in late timeout. | No hard 90s force-resolve guarantee in both paths today. **NOT IMPLEMENTED — needs Phase 2** (single SLA controller for 10s/45s/90s). | "This is taking longer than expected. We returned what we could." | `web/app/dashboard/clients/page.tsx`, `web/components/ApplicationUpload.tsx` |
| Scanned/image PDF with poor or no OCR text | Extraction returns mostly null/low-signal fields; no explicit OCR quality detector or scanned-file classifier in current path. | No dedicated scanned-doc fallback path beyond generic parse attempts. **NOT IMPLEMENTED — needs Phase 2** (quality classifier + partial-first UX). | "This file was hard to read. Please review all fields." | `web/lib/application-extractor.ts`, `web/lib/ingestion-v3-processor.ts`, `web/app/dashboard/clients/page.tsx`, `web/components/ApplicationUpload.tsx` |
| File is not a valid PDF | Client checks `file.type` or `.pdf` extension and direct parser route checks `File` + MIME/extension; non-PDF rejected with 400. | Immediate reject (no fallback). | "This doesn't appear to be a PDF. Please upload a PDF application." | `web/app/dashboard/clients/page.tsx`, `web/components/ApplicationUpload.tsx`, `web/app/api/parse-application/route.ts` |
| File exceeds size limit (single upload) | Add Client and direct parser enforce 13MB in code; upload-url backend allows up to 16MB (`MAX_UPLOAD_BYTES`). Mismatch can reject late or inconsistently. | Immediate reject where enforced; no cross-path standardization. **NOT IMPLEMENTED — needs Phase 2** (single 16MB policy everywhere per locked decision). | "This file is too large. Please upload a file under 16 MB." | `web/app/dashboard/clients/page.tsx`, `web/components/ApplicationUpload.tsx`, `web/app/api/parse-application/route.ts`, `web/app/api/ingestion/v3/upload-url/route.ts` |
| Network failure during upload | Browser `fetch` throws `TypeError` or timeout while PUT or API calls are in progress. | ApplicationUpload retries transient PUT failures (up to 3 attempts); Add Client generally does not retry and only conditionally falls back. **NOT IMPLEMENTED — needs Phase 2** (shared retry + fallback policy). | "Network issue while uploading. Please retry." | `web/components/ApplicationUpload.tsx`, `web/app/dashboard/clients/page.tsx` |
| Source file missing at job creation | `/api/ingestion/v3/jobs` checks `gcsPathExists(...)` and returns `UPLOAD_NOT_FOUND` (404). | No automatic re-upload/repair path. **NOT IMPLEMENTED — needs Phase 2** (single retry to regenerate signed URL and re-upload). | "We couldn't find the uploaded file. Please upload again." | `web/app/api/ingestion/v3/jobs/route.ts` |
| Cloud Tasks auth/config environment failure | `cloud-tasks.ts` throws missing env/credentials/private key id or access token errors while enqueuing. | Route currently returns enqueue failure. No transparent direct parse in server route. **NOT IMPLEMENTED — needs Phase 2**. | "Processing service is temporarily unavailable. Please try again." | `web/lib/cloud-tasks.ts`, `web/app/api/ingestion/v3/jobs/route.ts` |
| Processor webhook auth mismatch | Process route returns `TASK_AUTH_INVALID` if `x-cloudtasks-webhook-secret` missing/mismatch. | No automatic fallback from this server-side failure to direct parse for in-flight jobs. **NOT IMPLEMENTED — needs Phase 2**. | "Processing service couldn't verify this request. Please retry." | `web/app/api/ingestion/v3/jobs/[jobId]/process/route.ts` |

## Telemetry Events Needed

| Event Name | When It Fires | Properties to Include |
|---|---|---|
| `application_upload_started` | User selects PDF and processing begins (both Add Client + ApplicationUpload). | `entry_point` (`add_client` \| `application_upload`), `file_size_bytes`, `file_extension`, `source` (`local`) |
| `application_upload_signed_url_failed` | Signed URL API fails or times out. | `entry_point`, `error_code`, `http_status`, `timeout`, `attempt` |
| `application_upload_put_failed` | GCS PUT fails after retries/timeouts. | `entry_point`, `http_status`, `timeout`, `attempt_count`, `error_category` |
| `application_job_create_failed` | `/api/ingestion/v3/jobs` fails (including `TASK_ENQUEUE_FAILED`, `UPLOAD_NOT_FOUND`). | `entry_point`, `error_code`, `http_status`, `idempotency_key_present` |
| `application_poll_stalled` | Status remains queued/processing past stall cutoff. | `entry_point`, `job_id`, `stalled_status`, `stalled_ms` |
| `application_fallback_triggered` | Any automatic fallback to direct parse is invoked. | `entry_point`, `trigger_reason` (`queue_stall` \| `timeout` \| `processor_failed` \| `signing_error`), `job_id` |
| `application_fallback_failed` | Direct parser fallback fails or times out. | `entry_point`, `error_code`, `http_status`, `timeout` |
| `application_parse_completed` | Final terminal outcome emitted to UI. | `entry_point`, `result_type` (`full` \| `partial` \| `error`), `elapsed_ms`, `used_fallback` |
| `application_core_completeness` | Extraction result available (full or partial). | `entry_point`, `job_id`, `core_fields_total`, `core_fields_present`, `completeness_ratio`, `carrier_detected` |
| `application_sla_breach` | Any SLA threshold is breached (10s progress, 45s usable target, 90s hard max). | `entry_point`, `breach_type`, `elapsed_ms`, `current_stage`, `job_id` |
